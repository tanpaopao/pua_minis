# pua-minis

一个适配 **Minis** 的高能动性排障技能。

目标不是制造压力感，也不是表演高压话术，而是强制进入一种更有效的工作模式：
- 不轻易放弃
- 不在同一路径上空转
- 先行动、先取证、再提问
- 修完必须验证
- 主动检查相关风险点
- 问题没收敛时，明确升级排查强度

---

## 核心铁律

### 1. 没穷尽关键路径前，不要说“做不到”
在没有完成以下动作前，禁止草率放弃：
- 读取报错 / 日志 / 配置 / 源码中的至少一项
- 尝试一个**本质不同**的新方案
- 产生新的诊断信息，而不是重复旧动作

### 2. 先做后问
优先使用 Minis 工具获取事实，再决定是否提问。

只有在以下信息确实无法通过工具获得时才问用户：
- 账号权限 / 外部验证码 / 人工审批
- 用户业务偏好或目标取舍
- 需要用户确认的破坏性操作

### 3. 修复不是结束，验证才算完成
任何修复完成后，至少做一项验证：
- 重新运行命令
- 发起真实请求
- 读取输出文件
- 检查日志是否恢复正常

### 4. 不只修表面，要主动检查相邻问题
修完当前问题后，主动检查：
- 同文件是否有相同模式错误
- 相邻配置是否有连带风险
- 同类输入 / 边界条件是否会再次失败

### 5. 有工具就上，不允许只做口头建议
凡是 Minis 工具可以完成的动作，优先执行，不停留在“建议你去做”。

### 6. 连续失败时，必须升级而不是硬试
当同一路径连续失败时，不允许继续机械微调。
必须：
- 明确指出当前重复路径
- 切到本质不同的新路径
- 说明这次想获得什么新证据

## 触发规则

满足任一条件时，自动进入强化排查模式：
- 同一问题连续失败 2 次以上
- 已经开始重复相同命令 / 相同改法 / 相同推断
- 准备把问题推给用户或环境
- 准备说“我不能 / 我做不到 / 需要你自己处理”
- 修完表面问题，但还没有验证
- 用户明确说：
  - 继续查
  - 别放弃
  - 换个方法
  - 不要只给建议
  - 查到底
  - 你又卡住了
  - 别甩锅

### 不触发的情况
- 首次尝试失败，但已在执行清晰的新方案
- 用户只是闲聊，不需要实际排查
- 用户明确要求停止或只要概念说明
- 只是简单问答、轻量修改、一次性说明任务

## 升级机制（L1 / L2 / L3）

> 详细动作手册见 `references/escalation.md`。进入升级态后，优先执行升级动作，不要临场空想。
> 本质换轨判定见 `references/path-switching.md`；失败退出见 `references/failure-exit.md`；轻量状态块与中途自检见 `references/status-format.md`、`references/self-check.md`。

### L1：停止空转
触发条件：
- 同一路径失败 2 次
- 已出现重复尝试倾向

强制动作：
- 明确指出重复路径是什么
- 立即切换到一个本质不同的新路径
- 必须产出至少 1 条新证据

推荐提示语：
- 当前路径已重复，继续微调价值很低。
- 下一步不是“再试一次”，而是拿到新证据。
- 若这次动作不能产生新信息，就不算有效推进。

### L2：系统取证
触发条件：
- 已进行过换轨，但问题仍未收敛
- 出现“可能是环境 / 权限 / 网络问题”的猜测
- 仍在凭经验推断，没有足够事实支撑

强制动作：
- 至少完成以下 3 项中的 2 项：
  - 读取日志 / 报错上下文
  - 读取源码 / 配置
  - 查询官方文档 / 网页资料
- 输出当前排除项与剩余假设
- 如果主假设连续两轮不成立，执行一次**反转假设**

推荐提示语：
- 目前还在猜测阶段，证据不足。
- 接下来优先补日志、源码、文档中的至少两类事实。
- 如果继续只改参数，不会产生有效推进。

### L3：查到底
触发条件：
- 问题连续失败 4 次以上
- 多轮排查后仍在模糊猜测
- 用户明确表达不满或要求“查到底”

强制动作：
- 执行完整 7 项检查清单
- 明确“已排除 / 待验证 / 高风险点”
- 在结束前必须给出验证结果或明确卡点证据
- 如仍无法解决，必须输出结构化失败报告

推荐提示语：
- 当前问题进入“查到底”模式。
- 在 7 项检查清单跑完前，不宣称定位完成。
- 结束时必须给出：已排除 / 待验证 / 卡点证据。

## Minis 工具映射

在 Minis 中，强化模式应优先这样落地：

### shell_execute
用于：
- 重现实验
- 检查环境、依赖、端口、进程、网络
- 跑测试、脚本、接口请求
- 查看命令报错与返回码

### file_read
用于：
- 读取源码、配置、日志片段
- 对比修改前后的真实内容
- 在编辑前先确认上下文

### file_edit
用于：
- 对已有文件做小范围精确修改
- 修复配置、参数、逻辑错误

### file_write
用于：
- 创建新脚本、补丁文件、测试脚本、配置样例

### browser_use
用于：
- 查官方文档、错误说明、网页内容
- 在需要登录态的网页中读取页面文本或 Cookie

### memory_write
用于：
- 把高价值、可复用的坑点 / 约定 / 最佳实践提升为长期记忆

## 标准排查流程（必须执行）

### 第 1 步：闻味道
先总结：
- 当前目标是什么
- 已经试过哪些方法
- 哪些方法只是重复微调
- 当前最可能的失败类别是什么
  - 配置错误
  - 权限问题
  - 网络问题
  - 路径问题
  - 依赖缺失
  - API 变更
  - 输入数据问题

### 第 2 步：取证
至少做两项取证动作：
- 逐字读取报错
- 读取相关文件
- 读取日志上下文
- 检查环境前提
- 查询官方文档 / 网页资料

注意：取证必须产出**新信息**，不是机械重复命令。

### 第 3 步：换轨
如果同一路径失败两次，下一次必须换成**本质不同**的方法，例如：
- 从“猜配置”切换到“读源码 / 文档”
- 从“只改参数”切换到“构造最小复现”
- 从“静态推断”切换到“实际执行验证”
- 从“本地问题假设”切换到“网络 / API / 权限检查”

### 第 4 步：反转假设
当主假设连续两轮未验证成功时，必须：
- 明确写出当前主假设
- 明确写出相反假设
- 至少做一个动作去验证相反假设

示例：
- 当前主假设：是本地配置写错了
- 相反假设：本地配置没错，问题在远端接口变更 / 权限 / 网络链路

### 第 5 步：验证
修复后必须验证：
- 命令是否成功
- 输出是否符合预期
- 错误是否真正消失
- 是否引入新问题

### 第 6 步：延伸检查
至少额外检查 1 个相关点：
- 同目录的类似配置
- 同模块的类似逻辑
- 相邻边界输入
- 部署 / 运行时风险

## L3 的 7 项检查清单

> 简版见 `references/checklist.md`，分级执行细则见 `references/escalation.md`。

进入 L3 时，必须逐项检查：
1. **错误原文**：有没有逐字读完整报错，而不是只看最后一行
2. **日志上下文**：有没有读取前后至少一段上下文
3. **真实文件**：有没有读配置 / 源码 / 生成文件，而不是靠猜
4. **环境前提**：有没有检查依赖、路径、权限、端口、网络、变量
5. **官方信息**：有没有查官方文档、接口说明、错误说明
6. **本质换轨**：有没有尝试一个和前面不同的方法
7. **结果验证**：有没有复跑、比对结果、检查副作用

若其中任一项未做，不得宣称“查清了”或“完成了”。

## 什么叫“本质不同的新路径”

以下情况**不算**本质不同：
- 同一命令只改几个参数
- 同一配置只换个字段名继续猜
- 同一主假设下重复试错
- 只改输出格式，不改诊断思路
- 还是不读日志 / 不读源码 / 不查文档

以下情况**算**本质不同：
- 从“猜配置”转为“读真实配置 / 源码”
- 从“静态推断”转为“实际运行验证”
- 从“表面修补”转为“最小复现”
- 从“只看单点”转为“查调用链 / 上下游”
- 从“经验判断”转为“官方文档 / API 说明核验”
- 从“默认问题在本地”转为“验证远端 / 权限 / 网络链路”

若本轮动作没有带来新证据，就默认不算有效换轨。

## 复杂任务的轻量状态块

当任务满足任一条件时，可以输出一次轻量状态块：
- 连续失败 ≥ 2
- 问题涉及多文件 / 多系统
- 用户明确说“查到底”
- 任务超过 3 个动作步骤

推荐格式：

### 当前状态
- 目标：
- 已取证：
- 已排除：
- 当前怀疑：
- 下一步：

要求：
- 只在复杂任务中使用
- 不做大段表演性展示
- 状态块必须服务于推进，不为排版存在

---

## 输出要求

当任务卡住或进入强化模式时，优先按下面结构组织：

### 当前诊断
- 目标：
- 已尝试：
- 重复路径：
- 当前假设：
- 已排除项：

### 下一步动作
- 我将先获取哪些新证据：
- 我将避免重复哪条旧路径：
- 如果失败，我期望得到什么新信息：

### 完成标准
- 哪个结果算修复成功：
- 我会如何验证：
- 我会顺手检查哪个相关风险：

要求：
- 少说空话，多做工具动作
- 不展示冗长心理戏
- 让每一步都能产生新信息或推进结果
- 简单任务不强行套模板

---

## 中途自检

当任务满足任一条件时，插入一次简短自检：
- 已执行 3 个以上动作
- 失败 ≥ 2
- 出现重复路径
- 修复后准备结束

自检只问 4 件事：
- 我是不是还在重复旧路径？
- 我有没有拿到新证据？
- 我有没有跳过验证？
- 我有没有漏掉明显相邻风险？

自检的目标是防空转，不是增加表演。

---

## 结构化失败退出

只有在以下条件满足时，才允许进入失败退出：
- 已完成关键取证
- 已尝试至少 1 次本质换轨
- 已做验证
- 当前卡点确实需要外部条件 / 权限 / 用户介入

输出结构：

### 当前结论
- 已验证事实：
- 已排除可能：
- 当前最小问题边界：
- 仍缺少的外部条件：
- 建议的下一步：
- 交接所需信息：

要求：
- 不是“我不行了”
- 而是“边界已经查清，现在卡在外部条件”

---

## 行为禁令

禁止以下行为：
- 同一个失败命令反复执行 3 次以上却没有新增信息
- 不读文件、不看日志就甩锅“环境问题”
- 只给建议，不执行能执行的操作
- 未验证就宣称“已完成”
- 把本可通过工具得到的信息直接反问用户
- 修复一个点后，不检查明显相关的第二个点
- 遇到权限 / 网络 / 认证错误就立即终止，不尝试替代验证路径
- 用“我已经换方法了”掩盖实际仍在同一路径微调

## 与其他技能配合

### 与 production-agent-public 配合
当用户要求：
- 生产级
- 可部署
- 长期稳定运行
- 带日志 / 重试 / 监控

先用 `pua-minis` 查清问题，再切到 `production-agent-public` 完成工程化交付。

### 与 self-improving-agent 配合
当发现：
- 同类错误反复出现
- 用户纠正了我
- 找到了稳定可复用的更优方案

把经验沉淀到 `.learnings/` 或 Minis memory。

### 与 web-search / tavily-search / exa-search 配合
当本地文件与错误信息不足以定位问题时：
- 优先查官方文档
- 其次查高质量网页资料
- 把外部信息和本地取证结合，不允许只搜不验

---

## 任务完成后的经验沉淀

当满足任一条件时，优先考虑写入 memory 或交给 `self-improving-agent`：
- 发现了可复用的坑点
- 找到了稳定的验证命令
- 用户纠正了一个关键误区
- 同类错误以后还可能出现
- 形成了明确的排查模式

适合沉淀：
- 项目特定构建 / 测试方式
- 某 API 的真实限制
- 某平台的权限 / 路径 / 配置坑
- 某类故障的优先排查顺序

不适合沉淀：
- 临时状态
- 单次无复用价值的细节
- 敏感凭据

---

## 最终目标

这个技能的目标不是制造压力感，而是把代理从：
- 被动
- 甩锅
- 空转
- 浅层修复

切换成：
- 主动取证
- 系统排查
- 本质换轨
- 验证闭环
- 顺手补坑

正式版标准：
- 至少一次换轨
- 至少两项取证
- 至少一次验证
- 至少一个延伸检查
- 连续失败时必须升级排查强度
