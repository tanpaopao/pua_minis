# pua-minis 升级动作手册

## 目标

当任务进入反复失败、空转、甩锅、浅层修复等状态时，使用本手册把排查动作升级为可执行的标准流程。

---

## L1：停止空转

### 触发条件
- 同一路径失败 2 次
- 已经出现重复命令 / 重复改法 / 重复推断
- 开始倾向于口头解释，而不是实际取证

### 目标
停止重复，强制获得第一条新证据。

### 必做动作
1. 明确写出“重复路径”是什么
   - 重复执行了什么命令
   - 重复改了什么参数/配置
   - 重复依赖了什么假设
2. 立刻换成一个本质不同的新动作
3. 至少产出 1 条新信息

### 推荐动作
- 读报错完整原文
- 读相关文件而不是继续猜
- 构造最小复现
- 用真实执行代替静态推断

### 通过标准
满足以下任意两项即可视为 L1 完成：
- 找到新的错误细节
- 排除一个旧假设
- 获得一条可验证的新方向

### 失败信号
如果换轨后仍然只得到模糊猜测，升级到 L2。

---

## L2：系统取证

### 触发条件
- 已做过一次换轨，但问题仍未收敛
- 出现“可能是环境 / 网络 / 权限 / API”的模糊猜测
- 需要把问题从“猜”转成“证据驱动”

### 目标
建立证据链，缩小问题范围。

### 必做动作
至少完成以下 3 项中的 2 项：
1. 读取日志 / 报错上下文
2. 读取源码 / 配置 / 生成文件
3. 查询官方文档 / 错误说明 / 变更信息

并且输出：
- 已排除项
- 待验证项
- 当前最高概率假设

### 推荐动作
- 查看命令返回码与 stderr
- 检查环境变量是否存在（只判断 set / not set，不输出值）
- 检查依赖版本、路径、端口、权限
- 对照官方文档核验参数/接口

### 通过标准
满足以下任意三项即可视为 L2 完成：
- 已明确排除至少 2 个假设
- 问题范围明显缩小
- 获得具体可执行的下一步修复动作
- 找到与本地事实或官方资料一致的根因线索

### 失败信号
如果已经做了系统取证，仍然没有明确方向，或用户明确要求“查到底”，升级到 L3。

---

## L3：查到底

### 触发条件
- 问题连续失败 4 次以上
- 多轮排查后仍在模糊猜测
- 用户明确说“查到底”“别放弃”“不要甩锅”
- 该问题影响关键交付或生产可用性

### 目标
执行完整检查闭环，不允许停在半证据状态。

### 必做动作
必须逐项执行 7 项检查：
1. 读完整错误原文
2. 读日志上下文
3. 读真实文件
4. 查环境前提
5. 查官方信息
6. 做本质换轨
7. 做结果验证

### 输出要求
必须明确给出：
- 已排除项
- 待验证项
- 当前最可能根因
- 如果仍未完全解决，卡点证据是什么

### 推荐动作
- 重新梳理全部尝试路径，标记重复和无效尝试
- 检查隐藏日志目录、缓存、临时文件、生成物
- 从“调用方”切到“被调用方”，或反向验证
- 读官方 changelog / issue / 文档说明
- 构造最小输入与最小环境做隔离验证

### 通过标准
只有满足以下条件才可结束 L3：
- 有明确验证成功结果；或
- 有明确、可复述、可证据支持的卡点说明

### 禁止结束的情况
以下情况不得结束：
- 只有猜测，没有证据
- 只说“可能是环境问题”
- 只说“建议你手动检查”
- 修了一个点，但没复跑验证

---

## L1 / L2 / L3 的核心区别

### L1
重点是：**停止重复，先拿到新信息**

### L2
重点是：**把模糊猜测变成证据链**

### L3
重点是：**把排查做完整，直到能验证或能证明卡点**

---

## Minis 工具建议映射

### 在 L1 优先用
- `shell_execute`
- `file_read`

### 在 L2 优先用
- `shell_execute`
- `file_read`
- `browser_use`

### 在 L3 优先用
- `shell_execute`
- `file_read`
- `file_edit` / `file_write`
- `browser_use`
- `memory_write`（用于沉淀高价值经验）

---

## 收尾动作

任何级别完成后，都建议补做：
1. 结果验证
2. 相邻问题检查
3. 如有复用价值，沉淀到 memory 或 `.learnings/`
