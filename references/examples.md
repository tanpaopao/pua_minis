# pua-minis 实战案例

## 案例 1：API 返回 401

### 常见错误做法
- 直接猜 token 过期
- 反复重试同一个请求
- 让用户“自己去后台看看”

### pua-minis 做法
1. 读完整报错与响应体
2. 检查认证相关环境变量是否 set（不输出值）
3. 对照官方文档核对请求头、认证方式、参数名
4. 区分：凭证缺失 / 凭证过期 / 权限不足 / 签名错误
5. 修复后重新发起真实请求验证

### 延伸检查
- 其他接口是否也共用同一认证逻辑
- 重试机制是否会把 401 当成可重试错误

## 案例 2：配置文件改了还是不生效

### 常见错误做法
- 继续重复改同一个配置项
- 猜测是缓存问题但不验证
- 没确认程序实际读的是哪个文件

### pua-minis 做法
1. 读真实配置文件内容
2. 检查程序启动参数、工作目录、默认配置路径
3. 查日志看最终加载的是哪份配置
4. 对比“修改的文件”和“实际生效的文件”是否一致
5. 修改后重启或重新执行，再验证效果

### 已验证自测
本地自测目录：`/var/minis/workspace/pua_minis_test/`

- 用户修改文件：`app.env`
- 程序最初实际读取：`default.env`
- 表面现象：配置改了不生效
- 真实根因：程序读错了配置文件，不是配置值没有修改成功

自测动作链：
1. 先运行脚本复现，输出仍为旧值
2. 读取脚本与两份配置文件
3. 发现脚本读取的是 `default.env`，不是 `app.env`
4. 修复脚本读取路径
5. 重新运行，确认输出变为新值
6. 额外检查目录内是否还有其他错误配置引用

自测结果：该案例已通过真实文件 + 真实执行验证，说明本 skill 在“配置改了不生效”场景下能够完成：
- 先复现
- 读真实文件
- 锁定根因
- 修复并验证
- 延伸检查

### 延伸检查
- 是否存在环境变量覆盖配置文件
- 是否有多环境配置（dev/test/prod）混淆

## 案例 3：端口被占用导致服务启动失败

### 常见错误做法
- 只看到 bind failed 就直接改端口
- 不确认是哪个进程占用
- 改完端口后不检查上下游引用

### pua-minis 做法
1. 读取完整错误原文
2. 用命令检查端口占用与对应进程
3. 判断是旧进程未退出、其他服务占用，还是权限限制
4. 选择方案：停止旧进程 / 改端口 / 更新配置引用
5. 重新启动并验证监听状态

### 延伸检查
- 反向代理、健康检查、客户端配置是否也要同步改端口
- 是否有僵尸进程或重复启动脚本

## 案例 4：依赖明明装了但运行时报缺失

### 常见错误做法
- 一直重复安装同一个包
- 猜 pip 坏了、环境坏了
- 不区分系统 Python / 虚拟环境 / uv 环境

### pua-minis 做法
1. 检查当前执行器到底是谁（python / python3 / uv run / venv）
2. 检查依赖安装位置和实际运行环境是否一致
3. 读取报错中的模块名与导入路径
4. 若有需要，构造最小导入测试
5. 修复后实际运行目标命令验证

### 延伸检查
- requirements / pyproject / lock 文件是否一致
- 是否存在多个解释器混用

## 案例 5：网页抓取失败或页面内容异常

### 常见错误做法
- 一直请求同一个 URL
- 看到空页面就说网站封了
- 不检查登录态、重定向、UA、懒加载

### pua-minis 做法
1. 确认真正失败的是网络、鉴权、页面结构，还是渲染方式
2. 用 shell 检查 HTTP 响应，再用 browser_use 看页面文本
3. 检查是否跳转到登录页、验证页或错误页
4. 必要时改用官方接口、Cookie、浏览器会话或不同提取路径
5. 抓取后验证结果内容是否完整、是否是目标页面

### 延伸检查
- 页面是否是 CSR / SSR / 无限滚动
- 是否需要额外 headers、cookies 或延迟加载策略
