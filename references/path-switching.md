# pua-minis 本质换轨判定

## 目的

很多“换方法”只是表面换壳，实质仍在同一路径微调。
本文件用于判断：什么叫真正的本质换轨，什么不算。

---

## 不算本质不同

以下动作默认不算真正换轨：
- 同一命令只改几个参数继续试
- 同一配置只换一个字段名继续猜
- 同一主假设下重复小修小补
- 只调整输出格式，不调整诊断思路
- 继续不读日志 / 不读源码 / 不查文档
- 只是把“再试一次”包装成“新方案”

如果动作没有带来新证据，通常就不算有效换轨。

---

## 算本质不同

以下动作通常算真正换轨：
- 从“猜配置”转为“读真实配置 / 源码”
- 从“静态推断”转为“实际运行验证”
- 从“表面修补”转为“最小复现”
- 从“单点修复”转为“查调用链 / 上下游”
- 从“经验判断”转为“官方文档 / API 说明核验”
- 从“默认问题在本地”转为“验证远端 / 权限 / 网络链路”
- 从“只看最后一行报错”转为“读完整错误与上下文”

---

## 自检问题

在说“我已经换方法了”之前，先问自己：
1. 这次是不是改变了证据来源？
2. 这次是不是改变了主假设？
3. 这次是不是改变了验证方式？
4. 这次有没有产出前一轮没有的新信息？

如果 4 个问题里大多数都是否，说明大概率还没真正换轨。

---

## 常见换轨示例

### 配置类故障
- 不算：继续猜是哪一项配置写错
- 算：直接读取真实配置文件、启动参数、最终生效配置

### API 类故障
- 不算：重复改 header / timeout / 参数顺序
- 算：查官方文档、构造最小请求、验证鉴权链路

### 依赖类故障
- 不算：反复安装同一个包
- 算：检查解释器、虚拟环境、导入路径、运行入口是否一致

### 部署类故障
- 不算：重复重启服务
- 算：看日志、查端口、查权限、查启动命令与配置注入
